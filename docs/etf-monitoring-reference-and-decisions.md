# ETF 主题监控工作台：参考项目、数据源与架构决策

更新日期：2026-07-24

本文记录 ETF 主题监控工作台的外部参考、采用范围、未采用范围和工程取舍。它是可审查的决策记录，不是项目链接清单，也不把模型内部推理过程当作设计依据。后续如果数据源、许可证或部署条件变化，应在本文追加变更原因。

## 1. 研究目标与边界

首个研究目标：

> 持续监控 A 股通信与半导体 ETF，识别美股半导体隔夜行情、行业新闻和政策变化对 A 股 ETF 的传导影响。

默认用户标的：

| 角色 | 标的 | 用途 |
|---|---|---|
| 核心、完整分析 | `515880.SS` 通信 ETF | A 股通信主题主观察标的 |
| 核心、完整分析 | `512480.SS` 半导体 ETF | A 股半导体主题主观察标的 |
| 同类比较、轻量信号 | `159995.SZ` 芯片 ETF | 比较同主题产品的相对强弱与流动性 |
| 美股驱动、轻量信号 | `SOXX`、`SMH` | 美国半导体 ETF 的行业方向 |
| 美股驱动、轻量信号 | `NVDA`、`TSM`、`AVGO`、`AMD`、`ASML` | 隔夜龙头、产业链和事件驱动 |

沪深 300、纳指 100 和美元人民币作为系统基准，不占用户十个自选位置。

当前产品边界：

- 只做研究、回测、解释和提醒，不连接券商，不执行交易。
- “实时”指带来源和时间戳、可验证的 5–15 分钟近实时监控，不包装成逐笔或交易所级行情。
- 数据不足时输出“不足以归因”，不让 Agent 用叙述替代证据。
- 付费全文不入库，只保存标题、允许保存的摘要、元数据和原文链接。

## 2. 选型原则

本项目不是再造一个通用量化平台，而是把少数成熟思想放进一套低成本、可维护的 ETF 工作台。

采用一个外部方案前，依次检查：

1. 它是否直接改善 ETF 主题监控，而不是只增加功能数量。
2. 是否能在 Cloudflare Worker、D1 和 GitHub Actions 的成本与运行限制内工作。
3. 是否有明确的来源时间、降级状态和复现信息。
4. 是否能在不复制整套重型前后端的情况下拆分使用。
5. 数据接口、内容和代码许可证是否允许当前使用方式。
6. 上游失效时，页面是否仍能给出清楚的降级或不可用状态。

由此形成三层运行结构：

| 层 | 运行位置 | 适合的任务 | 不放入这一层的内容 |
|---|---|---|---|
| 近实时采集 | Cloudflare Worker，每 5 分钟 | 轻量 HTTP 行情、来源健康、新闻发现、15 分钟规则信号 | pandas 重计算、模型训练、完整多 Agent 辩论 |
| 深度研究 | GitHub Actions / Python | ETF 结构、完整 Agent、较长历史、复杂指标、离线验证 | 高频轮询、长期驻留服务 |
| 产品与证据 | Pages Functions + D1 + 网页 | 设置、行情、新闻、事件、对话、run card、引用 | 密钥、券商执行、无来源的“实时”数字 |

## 3. 参考项目评审

### 3.1 HKUDS Vibe-Trading

参考：[HKUDS/Vibe-Trading](https://github.com/HKUDS/Vibe-Trading)

值得吸收：

- Research Goal：先固定要研究的问题，再选择数据、工具和 Agent。
- 证据与假设的持久记录：保留数据源、状态、失效条件和后续观察点。
- run card：每次运行保存输入时间、工具、模型、降级、成本和产物，便于复查。
- 数据源自动降级：A 股可按 Tushare、mootdx、AKShare 等来源能力分层。
- 启动预检、持久记忆、SSE 进度和 MCP 工具的安全边界。

不整体并入：

- 它是面向多市场、回测、技能、MCP 和多 Agent 的完整研究平台，功能面明显大于当前 ETF 工作台。
- 整体合并会引入第二套前后端、运行时和大量技能注册，增加部署、升级和安全维护成本。
- 本项目已有 TradingAgents 分析内核和 Cloudflare/GitHub 运行链，重复引入会产生两套任务、会话和权限模型。

落地方式：

- 只移植 Research Goal、证据账本、run card、来源降级、启动预检的设计思想。
- A 股 Python 深度任务预留 `Tushare → mootdx → AKShare`；盘中 Worker 继续使用轻量 HTTP Provider。
- MCP 默认只读；触发深度分析作为受访问码保护的可选工具，默认关闭。

### 3.2 OpenBB

参考：[OpenBB-finance/OpenBB](https://github.com/OpenBB-finance/OpenBB)

值得吸收：

- “统一标准模型 + 可替换 provider”：调用方消费稳定 schema，来源适配器各自处理字段、认证和失败。
- 一个数据连接可以供 Python、REST、MCP 和研究界面复用。
- Provider 的能力和凭据是运行时配置，不写死在业务逻辑中。

不直接引入整个平台：

- 当前工作台只监控少量 ETF 和驱动标的，完整 OpenBB 运行时和扩展体系会放大依赖。
- 很多 provider 需要单独密钥、授权或商业订阅，不能假定“安装即有数据”。
- OpenBB 当前代码采用 AGPLv3；本项目只参考架构思想，不复制其受约束代码。

落地方式：

- 动态 API 固定返回 `{status, asOf, data, sources}`。
- 每条行情记录携带 `source`、`asOf`、`fetchedAt`、`freshness`、`adjustment` 和质量状态。
- Provider Registry 负责顺序、熔断和降级；指标、网页和 Agent 不依赖具体来源字段。

### 3.3 Microsoft Qlib

参考：[microsoft/qlib](https://github.com/microsoft/qlib)

值得吸收：

- Alpha158 等结构化因子表达与研究流程。
- IC、ICIR、基准超额收益等信号评价，而不是只看命中率。
- 交易成本、换手、最大回撤和实验记录。
- 时间切分、滚动验证和防止未来信息泄漏的研究纪律。

不放进五分钟 Worker：

- Qlib 适合离线数据集、模型、实验和回测，不适合边缘 Worker 的短时运行与轻量存储。
- 盘中监控首先需要新鲜、可解释、可降级的事实数据，而不是每五分钟重跑因子研究框架。

落地方式：

- 当前 Worker 只算规则信号和轻量指标。
- 后续在 GitHub Actions 的离线研究中增加 IC/ICIR、超额收益、成本、换手和回撤报告。
- 任何回测输出必须记录样本区间、基准、复权、成本假设和数据版本。

### 3.4 FinGPT

参考：[AI4Finance-Foundation/FinGPT](https://github.com/AI4Finance-Foundation/FinGPT)

值得吸收：

- 金融新闻实体识别、关系抽取、标题分类和情绪标注的数据任务拆分。
- 新闻与标的之间建立可审查的实体和关系，而不是只做关键词命中。

不直接采用“情绪预测涨跌”：

- 情绪是带模型误差的特征，不是交易结论。
- ETF 的驱动来自指数成分、政策、产业链、汇率和跨市场传导，单条标题情绪不足以解释价格。
- 中文缩写、ETF 代码和普通英文词可能误报，必须经过别名、上下文和回归用例约束。

落地方式：

- 新闻记录增加实体、相关度、方向、重要性、影响标的、主题和重复簇。
- 情绪只作为证据的一部分；最终结论同时检查价格异动、来源等级、时间衰减和反证。
- 为 `SMH` 等短缩写建立误报测试，短缩写不得脱离金融上下文单独匹配。

### 3.5 AI Hedge Fund

参考：[virattt/ai-hedge-fund](https://github.com/virattt/ai-hedge-fund)

值得吸收：

- 技术、情绪、风险等不同职责分别产生结构化意见，再由统一决策层汇总。
- 风险约束应由代码和数据保证，不能只靠提示词提醒。

不照搬：

- 该类项目常以知名投资者人物风格组织 Agent，展示性强，但不增加 ETF 数据质量。
- 个股基本面模板不适合直接套在 ETF 上；ETF 更需要指数、持仓、规模、流动性、费用、跟踪偏离和份额变化。

落地方式：

- 保留“技术—新闻—风险—综合”的职责分工。
- Agent 名称表达职责，不扮演人物。
- 最终报告采用“发生了什么—证据—可能传导—置信度—反证—下一观察点”。

### 3.6 Ashare 与 adata

参考：[mpquant/Ashare](https://github.com/mpquant/Ashare)、[1nchaos/adata](https://github.com/1nchaos/adata)

值得吸收：

- 免费 A 股接口在现实中会变更，多来源热备比单一来源更实用。
- 统一清洗后的行情结构能把上游变化隔离在适配器内。
- 来源切换必须自动化，也必须让调用方知道发生了降级。

使用限制：

- 免费网页接口没有交易所级服务承诺，字段和访问策略可能变化。
- 这类 Python 库更适合深度任务、数据补洞和离线校验，不适合直接塞进 Cloudflare Worker。

落地方式：

- 盘中 Worker 使用腾讯、东方财富、Yahoo 等轻量 HTTP adapter，并保留来源健康与熔断。
- Python 深度链再使用 Tushare、mootdx、AKShare、adata 等可替换 adapter。
- 任一来源连续失败三次暂停十五分钟；页面显示实际来源和降级原因。

### 3.7 awesome-systematic-trading

参考：[wangzhe3224/awesome-systematic-trading](https://github.com/wangzhe3224/awesome-systematic-trading)

它适合作为能力地图，不作为运行依赖。最有价值的是：

- SEC EDGAR、公司公告和宏观数据入口。
- 经济日历、风险、期权和组合分析工具目录。
- 研究、回测、执行之间的边界示例。

不采用的部分：

- 券商路由、自动下单和实盘执行超出本项目范围。
- “awesome” 清单只是发现入口，选中的来源仍需检查官方文档、数据许可、更新时间和失败语义。

### 3.8 QuantStats

参考：[ranaroussi/quantstats](https://github.com/ranaroussi/quantstats)

适合后续吸收的内容：

- 组合与基准收益对比。
- 最大回撤、波动率、Sharpe/Sortino、月度收益和滚动指标。

使用方式：

- 作为离线报告指标参考，不进入盘中采集链。
- 研究结果必须同时显示成本、基准和样本区间，避免用单一收益率美化结果。

## 4. 行情数据源设计

### 4.1 盘中 Worker

当前优先级：

| 市场 | 顺序 | 说明 |
|---|---|---|
| A 股 | 腾讯 → 东方财富 → Yahoo | 免费、轻量 HTTP；必须带来源状态与时间戳 |
| 美股 | Yahoo → 可选 Alpha Vantage → Stooq 日线 | Alpha Vantage 只在配置密钥时启用；Stooq 仅作尽力而为的日线兜底 |

规则：

- 一次请求只接受 schema 校验后的 OHLCV。
- 空值、时间错位、HTML challenge 或异常 content-type 都视为来源失败。
- 连续失败三次，暂停该来源十五分钟。
- 成功后重置失败计数。
- 5 分钟行情保存 90 天，日线保存 5 年。
- 过期数据不能继续聚合成 `ok`，也不能冒充实时。

### 4.2 Python 深度任务

建议顺序不是“永远固定”，而是能力与权限驱动：

1. Tushare：配置 token 且接口权限足够时优先。
2. mootdx：免 token 的 A 股直连补充。
3. AKShare / adata：覆盖更广，适合数据补洞和交叉校验。
4. 本地缓存：只能作为明确标记的 stale 降级，不伪装为最新数据。

### 4.3 统一数据契约

动态响应：

```json
{
  "status": "ok",
  "asOf": "2026-07-23T07:00:00.000Z",
  "data": {},
  "sources": []
}
```

`status` 只能是：

- `ok`：主来源有效且新鲜。
- `degraded`：使用了备选来源，或部分输入不可用但结果仍可解释。
- `stale`：数据存在但超过新鲜度阈值。
- `unavailable`：没有足够数据形成结果。

网页、指标和 Agent 都必须保留这些状态，不得在中间层抹平。

## 5. 新闻与事件源设计

### 5.1 证据层

这些来源可以支持“发生了什么”的事实陈述：

- 上交所、深交所基金公告。
- 巨潮资讯。
- 中证指数和基金管理人公告。
- 工信部、国家统计局等政府部门。
- SEC EDGAR。
- 上市公司 Investor Relations。
- Federal Reserve。

证据层也不是绝对正确，但具有可追溯的原始发布主体、时间和原文。

### 5.2 发现层

用于快速发现线索：

- GDELT。
- Google News RSS。
- Yahoo Finance。
- 东方财富、财联社。
- RSSHub 适配的公开源。

发现层规则：

- 聚合站不能被标成原始发布者。
- 发现层内容需要链接回原始公告，或在结论中明确标注“尚未找到原始证据”。
- 不复制付费全文。
- 来源数量不能代替独立证据数量；同一稿件的转载只算一个重复簇。

### 5.3 新闻处理管线

```text
采集
  → 规范化时间、来源和链接
  → ETF / 指数 / 基金公司 / 成分股实体别名匹配
  → 普通缩写误报过滤
  → 相似标题与同稿转载聚类
  → 来源等级、相关度、重要性和方向标注
  → 时间衰减
  → 与价格/成交量异动对齐
  → 检查反证
  → 生成可引用事件
```

价格与新闻“共振”只能提高事件优先级，不能证明因果关系。报告应使用“可能传导”“时间上吻合”等准确措辞，并列出其他可能解释。

## 6. Agent 与证据设计

### 6.1 固定输出

每个主题结论按以下顺序输出：

1. 发生了什么。
2. 证据及其时间。
3. 对 A 股 ETF 的可能传导路径。
4. 置信度和主要假设。
5. 反证或替代解释。
6. 下一观察点。

### 6.2 证据账本

一次研究运行至少记录：

- Research Goal 和 profile ID。
- 标的角色、分析深度和基准。
- 每条输入数据的 `asOf`、`fetchedAt`、来源和复权状态。
- 来源失败、熔断和降级。
- 使用的新闻、公告和去重簇。
- 指标版本、参数和数据窗口。
- 模型、耗时、token 用量。
- 已解决结论、未解决问题和反证。

### 6.3 Run card

Run card 是面向复查的运行摘要，不是 Agent 的自由文本报告。建议字段：

```json
{
  "runId": "run-...",
  "profileId": "semiconductor-etf",
  "goal": "...",
  "scheduledFor": "...",
  "startedAt": "...",
  "finishedAt": "...",
  "inputAsOf": {},
  "sources": [],
  "degradations": [],
  "indicatorVersion": "...",
  "models": [],
  "usage": {},
  "evidenceIds": [],
  "unresolvedQuestions": []
}
```

### 6.4 研究问答的可靠性设计

问答不是普通聊天窗口，而是证据账本的查询入口。当前采用以下约束：

- 浏览器为每次提问生成稳定的 `requestId`，同时提交 `sessionId`、profile、当前标的和历史摘要。
- D1 先原子领取 `requestId`，再调用模型。同一个 ID 的重复请求只回放已保存结果；相同 ID 如果被用于不同问题，直接返回冲突，不再调用模型。
- 服务端保存问题、完整回答、上下文哈希和响应元数据。浏览器本地历史损坏或设备刷新后，可以按会话 ID 从 D1 恢复。
- SSE 连接中断后，服务端继续消费已经开始的模型响应并写入 D1；网页使用原 `requestId` 查询恢复结果，不通过重新提问制造第二笔模型费用。
- 动态上下文将当前标的的 15 分钟行情、新闻和事件编号为 `[M1]`、`[N1]`、`[E1]`，每条记录保留来源、数据时间、质量和原文链接。
- 回答必须保留证据编号与数据时间。只有价格没有相关新闻，或只有新闻没有可对齐行情时，不允许把时间相关包装成确定因果。
- 上下文经过 SHA-256 哈希，便于核对同一回答实际使用的材料版本；哈希用于审计，不替代原始证据。
- 访问码只通过请求头发送。会话接口同样需要访问码，D1、网页和日志都不保存访问码或模型密钥。

没有 D1 时，问答仍可使用静态报告降级运行，但响应会标明持久化不可用；此时不能宣称支持服务端恢复或跨设备会话。

## 7. 已完成的架构决策

截至本文日期，已经落地：

- `WorkbenchSettingsV2`、默认 ETF 研究目标与 v1 兼容迁移。
- D1 中的设置主存储、行情、新闻、事件和来源健康数据模型。
- 行情、新闻、事件、监控状态动态 API。
- A 股和美股免费行情 Provider Registry、熔断、降级与过期数据处理。
- 五分钟定时 Worker、理论时间槽、幂等领取、失败重试、租约恢复和 attempt fencing。
- 05:35、08:25、盘中采集/信号、15:20 深度分析的调度框架。
- GitHub 深度分析触发所需的仓库、workflow 和 token 仅从运行环境读取。
- 连续式专业研究终端、十标的自选、分窗格 K 线/成交量/MA/MACD/RSI 和增量刷新。
- 网页任务编辑、运行状态、立即执行、来源降级和移动端分区。
- D1 对话会话、请求幂等、完成结果回放、断线后恢复和本地历史合并。
- 行情、新闻、事件证据编号、时间戳、上下文哈希和“证据不足不得归因”的问答约束。
- Cloudflare D1 `tradingagents-workbench` 已创建并完成全部迁移，默认 profile 只在空库时写入，不覆盖网页配置。
- Pages 项目 `tradingagents-board` 已部署问答与工作台版本；独立 Worker `tradingagents-monitor` 已启用五分钟 Cron，并绑定同一 D1。
- 动态行情 API 已统一计算 MA20/60、MACD 12/26/9、RSI14、ATR14 和 20 期实现波动率，并返回指标版本、复权状态和算法口径。
- 十五分钟盘中信号已从占位逻辑改为真实事件：按 15 分钟涨跌和近期成交量 z-score 分级，事件 ID 与计划时间槽绑定，重复 Cron 不会重复入库。

仍需完成或继续增强：

- 完整新闻证据管线和官方源 adapter。
- 跨市场相关性、隔夜传导与 ETF 结构分析。
- 只读 MCP 工具与受保护的深度分析入口。
- 生产数据持续采集观察和真实模型问答冒烟验收；本地 fixture 与部署成功状态不能代替回答质量验收。

## 8. 被明确拒绝的方案

| 方案 | 拒绝原因 |
|---|---|
| 整体合并 Vibe-Trading | 重复前后端、会话、权限和任务系统，维护面过大 |
| 整体安装 OpenBB 作为在线 Worker | 依赖和密钥面过大，不适合边缘短任务 |
| 每五分钟运行 Qlib 或完整 TradingAgents | 成本、时长和噪音都不合理 |
| 把情绪分数直接当涨跌预测 | 证据不足，容易将模型偏差包装成结论 |
| 用聚合新闻代替交易所或公司原文 | 无法稳定确认发布主体与事实版本 |
| 无来源时计算并展示 ETF 溢折价或 iNAV | 会制造看似精确但不可验证的数据 |
| 把旧缓存显示成“实时” | 破坏数据可信度 |
| 接入券商自动交易 | 超出研究工具范围和当前风险边界 |

## 9. 原有能力的兼容边界

ETF 工作台是 TradingAgents 上方的监控与编排层，不是一次重写。以下能力属于必须长期保留的产品契约：

- **期权监控独立保留。** 上证 50 期权风控继续由 `sh50-volguard` 独立页面承载，ETF 工作台桌面顶栏和移动端底栏都提供常驻入口；仓库内 `/api/volguard` 继续承担实时源失败后的快照降级。这样不会把期权特有的波动率、Greeks、到期结构和风控逻辑硬塞进 ETF 页面。
- **TradingAgents 核心不被替换。** `TradingAgentsGraph`、CLI、分析师/研究员/交易员/风险经理协作图、报告写入器、历史报告接口、定时分析和 Issue 点名分析流程继续存在。工作台只负责选择标的、触发 workflow 和读取结果。
- **深度与信号严格分开。** 十个监控标的用于五分钟行情与十五分钟轻量信号；网页“多智能体分析”只提交标记为 `analysis: full` 的核心标的，默认是 `515880.SS` 和 `512480.SS`，避免把八个比较/驱动标的错误地扩成完整辩论。
- **新旧入口同时可发现。** 桌面顶栏固定显示“多智能体分析”和“期权风控”，移动端保留“期权”入口；研究档案继续读取原有报告，不另建一套互不兼容的报告格式。
- **行情与系统状态分开用色。** 面向 A 股用户的行情统一采用红涨绿跌，K 线、成交量、MACD、涨跌数字和多空评级保持一致；数据源健康、任务成功和接口错误仍采用绿/黄/红状态色，避免把“上涨”与“系统正常”混为一个语义变量。
- **回归测试锁定契约。** 测试会同时检查期权 API 的实时/快照降级、核心图与 CLI 文件、分析 workflow、报告读取接口、网页入口及深度标的筛选。后续页面改版不能以“简化界面”为由删除这些能力。

部署上仍保持两个独立 Cloudflare Pages 项目：`tradingagents-board` 是 ETF 与多智能体研究工作台，`sh50-volguard` 是期权风控终端。两者在产品导航上连通，在运行和故障域上隔离；一边升级或降级不会让另一边整站不可用。

## 10. 后续决策检查表

新增数据源或参考项目时，需要补充：

- 原始项目或官方文档链接。
- 数据覆盖、更新时间、历史深度和复权方式。
- 免费额度、密钥要求和部署限制。
- 内容和代码许可证。
- 失败状态是否可识别。
- 是否支持来源时间和原文链接。
- 放在 Worker、Python 深度任务还是仅作研究参考。
- 替代来源和降级顺序。
- 对现有 schema、测试、成本和安全边界的影响。

任何来源在没有完成上述检查前，只能进入实验环境，不能进入生产证据链。
